# 电台 App 设计存档

> 本文档记录电台 App 的架构决策、数据结构、提示词与生成逻辑，便于跨会话延续开发。

## 一、整体架构（三层模型）

```
分类（Category）
  · 决定整体基调和播出时间
  · 预设 5 个 + 用户可自定义
  · 每个分类挂一个「标签池」
      ↓ 进入分类后「刷新」
台 / 节目预览（Station Preview）
  · AI 一次生成 N 个电台预览（堆叠卡片，左右切换）
  · 每个预览带「标签」（1-2 个，从该分类标签池里选）
  · 标签是功能性路由键：决定写作指导 + 数据注入 + 互动倾向
      ↓ 点选某个台「开播」
节目详情（Program Detail）
  · 按标签取对应「写作指导 guide」+「数据源 dataSource」生成完整节目
  · 叙述流格式（复用通话格式的变体）
  · 末尾按标签的互动倾向渲染互动 UI（投票/点歌/抽奖/连线）
```

## 二、首页 UI

- 两列网格卡片：每个分类一张卡（中间圆形图标=主题色底+白色 SVG+淡阴影，下面分类名 + 播出时段文案）
- 末尾两张特殊卡：随机频道、添加分类
- 遮罩（自定义壁纸）模式下卡片转半透明深色玻璃 + 白字（已适配）
- 进入分类 → 节目单页（空状态：ON AIR 灯灭 + 麦克风图标 + 调频收听按钮）

### 计划中的两 Tab（发现 / 我的）—— 暂缓，先跑通分类→台→详情
- 发现页：预设分类，探索新内容
- 我的页：收藏的台 + 自定义的台，长期订阅
- （注：后续讨论里"收藏的是台不是单期节目"，收藏共享同一份数据，靠 faved 标记）

## 五、标签（Tag）——核心配置

每个标签是「写作指导 + 数据注入 + 互动倾向」三件套：
```js
{
  id,                // 路由键，如 news_politics
  name,              // 显示名，如 时政要闻
  desc,              // 给生成预览的 AI 看，帮它判断填不填这个标签
  guide,             // 给生成详情的 AI 看，怎么写（重点工作量）
  dataSource,        // 可选：注入什么额外数据（无则纯生成）
  interactHint,      // 可选：互动倾向 none/vote/request/lottery/call
}
```

- 标签**按分类隔离**：每个分类自带标签池，随机频道时合并所有标签给 AI 随便挑。
- 标签价值有二：①路由写作指导 ②路由数据源（不同分类喂不同内容）。

### 预览输出字段（生成预览阶段，AI 填）
- 电台名称
- 主持人
- 嘉宾（可选，从关联 NPC 里选）
- 标签（选 1-2 个）
- 频道核心概念（150 字以上，描述节目概念与方向，**用户不可见**，喂给详情生成）
- 当前节目名
- 当前节目简介

## 六、资料注入（生成时喂给 AI 的上下文）

### 通用资料包 = `Phone._buildFullContext(opts)`
拼接顺序：
1. 世界观核心设定 = `Chat.getWorldviewPrompt()`（= world.setting + 通用货币，**仅核心总纲**）
2. 当前游戏时间（状态栏）
3. 世界观详细数据：
   - `npcBrief:true` → NPC 速查表（`NPC.formatQuickRef()`）+ 世界书角色简表
   - `npcBrief:false` → 全图 NPC 详细资料（全局 + 各地区/势力挂载）
   - 节日设定 + 知识设定（世界观 + 绑定世界书合并）
4. 用户面具信息
5. 单人卡主角 / 单人 NPC

> 注意：`getWorldviewPrompt()` 只是「基础资料」(setting+货币)；
> 「超详细版」= `_buildFullContext({npcBrief:false})` 的完整输出。

### 电台资料封装（计划）`_buildRadioContext({ tag, category, mainlineRounds })`
1. base = `_buildFullContext({ npcBrief })`  // 复用，不重抽
2. + 主线最近 N 轮（电台可能要 20 轮）
3. + 标签专属数据（按 tag.dataSource）：
   - 地方快报 → 当前地区详细资料 + 该地区势力/NPC 详细资料（按状态栏当前地区筛 region）
   - 热点圆桌 → 用户最近论坛帖
   - 路况电台 → 状态栏天气/位置
   - 随心点播 → 音乐播放器歌单
   - 娱乐头条 → 朋友圈/论坛八卦（待定）
4. + 标签 guide

## 七、输出格式（叙述流，复用通话格式的变体）

- **叙述行**：不加前缀。描写声音、垫乐、音效、主播语气、动作、演播室氛围。
- **台词行**：`> 说话人：内容`（带说话人名，**无时间戳**）。
- 节目级总时长在详情生成时由 AI 给一个字段（如 duration: "约30分钟"），显示在节目页顶部，不进正文。
- 计划新写 `_parseRadioReply`（不动通话的 `_parseCallReply`，两边需求已分叉：电台带说话人、无时间戳）。
- 渲染：台词气泡左对齐（单向广播，无"我方"），气泡上方小字标说话人名；叙述行=描述卡片。互动环节才可能出现玩家侧。

- NPC 资料用 `npcBrief:true`（速查表，播报不依赖 NPC 细节）
- dataSource：无
- interactHint：以纯播报为主（none），偶尔可做新闻话题投票（vote）

```
你正在生成「时政要闻」类电台节目。这是一档严肃正经的新闻播报，
主播口吻沉稳、客观、字正腔圆，像正式的晚间新闻联播。

内容方向（任选其一或组合，符合当前世界观）：
- 政策法规变动、经济数据与发展动态
- 国际关系、外交动向、国际局势
- 重大灾害事故及救援、应急通报
- 重要人物（领导人、重要官员）的公开动向与讲话
- 国家级文化/体育大型活动（类比春晚、奥运会级别）
- 反腐倡廉、扫黑除恶等专项行动通报

写法要求：
- 用"播报"语气，不是闲聊。每条新闻简明扼要，可分条播报多则。
- 信息要具体：给出地点、机构、数字、时间等细节，营造真实感。
- 紧扣当前世界观设定：现代都市就是现实新闻质感，古风世界
  就是朝堂邸报/告示，赛博世界就是全息政务播报，自动适配。
- 不编造与主线角色相关的剧情新闻，除非世界观/剧情明确支持。
- 保持中立客观，不带主播个人情绪和吐槽。
- 生成 3-4 条，每条新闻 400-500 字
- 时长：30 分钟左右
```

## 八点五、已实现机制（截至 phone.js v238）

### 标签池数据结构（已落地）
- 代码常量 `_RADIO_TAGS`（按分类 id 分组），不进 phoneData。
- 每标签：`{ name, desc, guide, dataSource, interactHint }`。
- 5 个分类共 31 个标签均已建好 name+desc；guide 仅时政要闻填了，其余留空（留空=走通用写法）。
- dataSource 已预埋：地方快报=region、热点圆桌=forum、路况电台=status、随心点播=playlist。
- `_radioTagsOf(catId)` 取分类标签池；自定义分类暂无标签（以后再说）。

### 预览生成（已接真实 AI，`_radioGenerate`）
- 复用项目现成的 `_phoneJsonArrayWithRetry`（JSON 数组+重试）。
- 喂料：`_buildFullContext({npcBrief:false})` 全量（已含主线最近10轮）+ 分类信息 + 标签名含义表 + 避让名单。
- 预览阶段**只给标签的 name+desc**，不注入各标签 guide/dataSource（标签此时由 AI 现挑，专属数据源留到详情阶段）。
- 输出字段：name(频道名) / dj / guest / tags / concept(隐藏,150字+) / showName(本期节目名) / showDesc / coverKeyword。
- **FM 前端生成**：`_radioGenFmList(n)` 生成 87.0–108.0 一位小数、**批内去重**（非全局），写死进 `p.fm` 跟台走。
- 封面：暂用预设 base64（`_RADIO_COVERS`，js/radio-covers.js）循环兜底；后续接 Unsplash 用 coverKeyword（用户填 key 才启用，逻辑同朋友圈配图）。
- 前端兜底：过滤非法标签、字段长度裁剪、空结果提示重试。

### 名字层级（重要）
- 卡片大标题 = **频道名 name**（台的长期身份，收藏/换台针对它）。
- showName = **本期节目名**（当期内容，会变）。
- 重新调频 = 换一批**新台**（name+showName 全换）。「查看频道更新」（订阅后，待做）= 同一个台**换新节目**（name 不变，只换 showName+正文）。

### 跨频道串台避让（已实现）
- 痛点：生成无状态，AI 爱让同一个活跃 NPC 跨分类反复出场。
- 方案：全局共享、软避让、滚动上限。`pd.radioRecentNpcs` 存最近出场真实 NPC 的 **id**（非名字），上限 15。
- 别名归一：`_radioBuildNpcIndex()` 把每个 NPC 的 name/aliases(别称)/onlineName(网名) 所有写法映射到同一 id；输出 `{aliasToId, idToName}`。
- 生成时：把 recentIds 翻译成主名拼进提示词软引导（"优先选名单外的人"）。
- 生成后：把命中的 dj/guest 经 aliasToId 归一成 id，更新进名单。
- 局限：AI 现编的、设定里没登记的马甲兜不住，只能靠提示词压。

### 详情页 UI（已做雏形，假正文驱动，`_radioOpenDetail`）
- 整页 overlay，挂进 `.phone-shell`（不是 body，否则定位跑飞）。
- 顶部栏：返回 + 台名/FM + 右上圆形头像（封面裁圆）。
- 滚动区：节目名/标签/主播嘉宾/总时长 → 正文 → 互动占位 → 结束语。
- 正文渲染 `_radioRenderSegments`：叙述行=居中圆角卡片；台词=气泡，主播用封面圆头像、嘉宾用固定话筒头像（紫底），气泡上方标说话人名，主播/嘉宾气泡底色区分。
- 解析器 `_radioReply`：`> 说话人：内容` 提说话人，无时间戳。
- 仍是 `_RADIO_FAKE_BODY` 假正文，**未接 AI**。

## 八点六、详情接真实 AI + 互动玩法系统（当前实现，截至 phone.js v398）

> 上面"八点五"是历史快照（v238 假正文阶段），下列为现状，以此为准。

### 详情正文已接真实 AI（`_radioGenProgramBody`）
- 已替换 `_RADIO_FAKE_BODY`，调功能模型（worldvoice 配置，回退主配置）生成整期正文。
- system 提示词拼装顺序：
  1. `_buildFullContext({npcBrief:true})` 全量资料（世界观/时间/NPC速查/节日/主线最近10轮）
  2. 历法块 `_radioCalendarBlock`（仅自定义历法）
  3. 地区资料块 `_radioRegionBlock`（仅 `dataSource==='region'` 的标签，命中当前地区才注入）
  4. 节目信息（电台名/本期节目名/FM/频道核心概念）
  5. 阵容块 `_radioCastBlock`（主播/嘉宾/念 FM 硬约束）
  6. 标签 guide（`_radioGuideOf`，空则走通用兜底文案）
  7. 互动玩法块 `_radioPlaysBlock`（标签挂了玩法才注入）
  8. 输出格式块 `_RADIO_FORMAT_SPEC`
- temperature 0.85，max_tokens 4000，生成后 `_stripMainlineArtifacts` 清洗。

### 互动玩法系统（plays 数组挂载）
- **玩法池 `_RADIO_PLAYS`**（key=玩法 id）。每个玩法字段：
  `{ id, anchor(正文锚点中文词), title, icon, descPending/Done/Skip(卡片三态文案), actLabel/skipLabel(按钮), handler(处理函数名), promptHint(注入正文生成时给AI的玩法说明) }`
- **目前有五个玩法**：
  - `mail`（读留言）锚点 `[[读留言]]` —— ✅ 已验证。
  - `vote`（投票）锚点 `[[投票|问题|选项A|选项B|选项C]]`（竖线分隔）—— ✅ 已验证。
  - `request`（点歌）锚点 `[[点歌]]` —— ⚠️ 尚未验证。
  - `call`（连线）锚点 `[[连线]]` —— ✅ 已验证（详见八点七）。
  - `lottery`（抽奖）锚点 `[[抽奖|奖品名称]]`（竖线分隔）—— ✅ 已验证（v571）。
- **标签挂载**：标签上加 `plays: ['mail', ...]` 数组，一个标签可挂多个玩法。当前挂载情况：社区简讯/领域专线=`['mail']`，娱乐头条/趣味新闻/狗血剧场/犀利锐评=`['mail','vote','lottery']`，随心点播=`['request']`，连线夜话=`['call']`。
- **固定玩法框架文案**（`_radioPlaysBlock` 生成，所有玩法共用）：告诉 AI——可选一个玩法、锚点必须在正文最末尾、自选触发时机（中段/尾声）、用了玩法就别自行收尾要留互动空间、不用就正常写完。
- **`_radioPlaysOf(prog)`**：合并节目所有标签的 plays，去重、过滤无效 id。

### 读留言玩法运行流程
1. AI 在正文末尾输出 `[[读留言]]`，`_parseRadioReply` 解析成 `{kind:'interact', interact:'mail', state:''}`（state 空=pending）。
2. 渲染成互动卡片，两个按钮：**我要留言**(play) / **不留言**(skip)。
3. 两条路径都调 `_radioGenMailSegment(prog, mail, priorBody)` 生成"读留言+收尾"段：
   - **我要留言**：弹窗收集玩家留言 → mail 有值 → 玩家留言必读（位置不限）+ 1-2 条虚拟听众。
   - **不留言**：mail=null → 全部虚拟听众（须与已播内容相关）。
   - 两者都带：同款全资料（含 region）+ `priorBody`(锚点前已播正文) + 读完承接主题收尾结束节目。
4. 生成段替换锚点为 `[[读留言:done]]\n{段落}`，续播。
5. **同步主线按钮**：正文有 pending 锚点时隐藏（`_radioHasPendingInteract`），互动完成后才显示。

### 固定下期预告（fixedNext）
- 标签可配 `fixedNext: { name(下期节目名), desc(预告语) }`，AI 不用填，显示在"我的"页下期预告块。
- 已配：时政要闻=晚间新闻、地方快报=本地新闻、社区简讯=邻里播报。
- 实例可用 `nextShow`/`nextPreview` 覆盖，否则回退标签 fixedNext（`_radioNextPreviewOf`）。

## 八点七、连线玩法 + 来电听众 + 动态下期预告（当前实现，截至 phone.js v499 / phone.css v187）

> 接续"八点六"。本节记录连线（call）玩法全链路、来电听众气泡、以及"AI 输出下期主题"的动态下期预告机制。emotion 分类的「连线夜话」是首个落地的连线类标签。

### 连线（call）玩法 —— 玩家与主播多轮实时连线
- 玩法定义在 `_RADIO_PLAYS.call`，锚点 `[[连线]]`，handler=`call`，promptHint 要求 AI 把连线安排在节目接近尾声、锚点必须是该环节最后一行（锚点后不能再有正文，后续由连线+收尾动态生成）。
- 标签挂载：连线夜话 `plays:['call']` + `connectMaterial:true`（标签专属随机素材开关）。
- **进入连线**（`_radioEnterCallMode`）：停朗读 → 隐藏底部播放器 → 原位插入连线输入条（状态提示"连线中" + 输入框 + 发送↑ + "等待主播回应" + 挂断）。
- **玩家发言**（`_radioCallSend`）：直接进滚动区右对齐气泡（`[[me:]]` 直显原文，不转述），可连发多条攒进 pending。
- **刷新打包**（`_radioCallRefresh`）：点"等待主播回应"才把 pending 批量请求主播回复（`_radioGenCallReply`），实现"呼吸感"对话。状态：刷新时"主播正在回应…"，完成回"连线中"。
- **实时回复**（`_radioGenCallReply`）：资料与节目正文同源（lite 速查表+历法+guide+castBlock+connectBlock+节目信息+已播正文+完整连线对话）；要求 2-5 句台词穿插 1-2 句声音叙述；**绝不替玩家说话**。
- **挂断**（`_radioCallHangup`，状态文案"通话结束，等待挂断…"）：把整通连线序列化进正文（玩家→`[[me:内容]]`，主播→原文台词行），替换 `[[连线:done]]` 锚点，再生成"收尾过渡段"（`_radioGenCallEnding`）接在连线之后。
- **挂断后不重播连线**：续播只从收尾段开始（`applyMailResult` 加 `opts.hideFromSeg/playFromSeg`），连线对话气泡保持可见、不再朗读；回放时整通连线能完整重现。
- **不连线（skip）**：走 `_radioGenCallSegment`，AI 现编 1-2 通虚构听众连线 + 收尾，资料同款。

### Lite 上下文（连线夜话专属瘦身）
- `_buildFullContext({ lite:true })` 物理跳过四块：NPC 列表与详细资料、玩家面具、住所、单人卡主角。
- 连线夜话相关生成函数统一用 `{ npcBrief:true, lite:true }`：只保留世界观基础 + 当前时间 + 节日 + 知识 + 速查表（NPC.formatQuickRef）+ 历法（仅自定义历法）。
- 地区详情不单独发，用速查表替代；历法 `_radioCalendarBlock` 仅在有自定义历法体系时返回内容。

### 来电听众气泡（caller，连线类通用）
- 解析器 `_parseRadioReply` 识别 `> 听众@化名：内容`（也兼容「来电@」「连线@」前缀）→ `{kind:'caller', speaker:化名, text}`。
- 渲染 `is-caller`：右对齐、装饰色气泡、电话听筒头像（`_radioCallerAvatar`）。CSS 同 is-guest 镜像（right-reverse），气泡用 `color-mix(decoration 12%)`。
- 与真玩家区分：真玩家本人走 `[[me:]]` 右对齐高亮气泡（永不替玩家代言）；虚构听众走 caller 气泡（AI 自拟匿名化名）。
- 格式说明抽成常量 `_RADIO_CALLER_FORMAT`，注入连线类的正文生成 + skip/挂断收尾段；连线夜话 guide 的【声音呈现】也改写成"听众用 caller 台词行"（旧版写死"听众一律叙述行转述"已纠正）。
- 实时回复气泡 `_radioCallAppendBubble` 同样识别 caller 前缀，并逐条 `seg-hidden→reveal` 延时浮现（主播多行错开 ~480ms/条），节奏对齐正文逐句弹出；叙述行 class 用 `phone-radio-narr`（与正文渲染器一致，修了早期误用 `phone-radio-desc` 导致裸奔的 bug）。

### 动态下期预告（[[next:]]，非新闻类通用）
- 替代固定 fixedNext：让本期 AI 在正文末尾输出元数据行 `[[next:下期节目名|下期简介]]`，由上一期"定"下一期主题。
- 格式常量 `_RADIO_NEXT_SPEC`：明确这是元数据（不渲染、不朗读、不是玩法钩子、不要写成台词），且下期主题须紧扣电台核心概念与标签调性。
- 解析器识别 `[[next:...]]` 行直接跳过（不入 segs）；`_radioExtractNext(raw)` 提取 `{name, desc}`。
- 注入范围：**所有非新闻类**。连线类正文以 `[[连线]]` 结尾、不能再有正文，其 next 改由连线收尾段（`_radioGenCallSegment`/`_radioGenCallEnding`）输出；其余非新闻类由正文末尾输出。新闻类保持原 fixedNext。
- 落库：`_radioPersistBody` 统一提取 next → 写入 `prog.nextShow/nextPreview`（覆盖正文生成、留言、连线挂断所有路径）。
- 续期消费（`updateBtn` 回调）：非新闻类 `showName ← nextShow`、`intro ← nextPreview`（**本期节目名/简介都来自上期产出的下期信息**），无 next 则清空（不沿用上一期标题/简介），消费后清空 nextShow/nextPreview 由本期重新产出。
- 正文生成时若 showName 为空，提示词明确告诉 AI"本期主题未定，请紧扣核心概念与调性自拟"。

## 十、2026.06.18 改动记录（v513-v521）

### 1. 面具泄露修复（v513）
**问题**：电台所有生成环节（预览、详情、留言、连线）未隔离玩家面具、住所、单人卡主角、挂载角色，会"泄露"玩家私人信息到节目内容。

**修复**：
- 所有电台生成函数统一用 `_buildFullContext({ npcBrief:true, lite:true })`
- `lite:true` 物理跳过四块：玩家面具、住所、单人卡主角、挂载角色
- 保留：世界观基础 + 时间 + 节日 + 知识 + NPC 速查表 + 历法（仅自定义）+ 主线最近 10 轮对话
- 影响范围：预览生成、详情生成、留言、连线实时回应、连线收尾段

### 2. 连线"发送面具"功能（v514）
**需求**：连线时可选择性告诉主播"我是谁"，让主播按关系相处。

**实现**：
- UI：连线输入条加圆形复选框"让主播知道我是谁（发送角色信息）"，默认不勾选
- 逻辑：点"等待主播回应"时读取勾选状态存入 `_radioCall.sendMask`
- 注入：勾选时调用 `_radioGetCallerIdentity()` 读取面具信息（姓名 + 网名 + 背景），单独拼成"【来电者身份】"块注入 sysPrompt
- 隔离：未勾选时保持 lite 模式隔离，勾选时单独注入面具块 + 主线 10 轮对话历史
- 清理：挂断后 `_radioCall` 对象清理，`sendMask` 字段自动消失
- 覆盖：实时回应（`_radioGenCallReply`）和中途收尾（`_radioGenCallSegment`）都接入此逻辑；挂断后最终收尾不读（连线已结束）

### 3. Guide 身份声明尝试与回滚（v515→v517）
**尝试**：批量将 11 个标签 guide 开头从"你正在生成「XX」类电台节目"改成"你是主播${djName}，你正在主持「XX」类电台节目"，强化身份认知。

**问题**：`_RADIO_TAGS` 是顶层常量定义，模板字符串里的 `${djName}` 在模块加载时立即求值，而此时 `djName` 未定义，导致整个模块加载失败，手机 App 打不开。

**回滚**：v517 回滚了 v515 的批量替换，guide 开头恢复成"你正在主持「XX」类电台节目"（去掉 `${djName}`）。

### 4. 占位符替换修复（v518）
**问题**：v515 回滚后 guide 里保留了 `$1` 占位符（`你正在主持「$1」类电台节目`），但 `_radioGuideOf` 函数返回 guide 时没有替换这个占位符，导致 AI 收到的是字面的 `$1` 而不是标签名。

**修复**：`_radioGuideOf` 返回时用正则把 `$1` 替换成标签名（`t.name`），确保 AI 能正确看到"你正在主持「犀利锐评」类电台节目"。

### 5. 犀利锐评标签强化（v516/v519/v520）
**目标**：提升犀利锐评的"网感"和"冷幽默"调性，让 AI 生成更犀利、更有梗的内容。

**改动**：
- **Guide 人设**（v516）：主播口吻改成"犀利、毒舌、充满各种冷幽默、善于吐槽，语气兼具阴阳怪气、反讽、各种网感搞笑发言。主播通常混迹豆瓣微博这类平台第一线，精神状态非常超前美丽，各种梗信手拈来。"
- **Desc 调性**（v519）：改成"充满阴阳怪气、反讽、搞笑吐槽、冷幽默的各类情感问题大吐槽"（影响预览生成时 AI 对标签的理解和 concept 生成）
- **写法约束**（v520）：加了"不要直接和任何投稿人对话，只以第三方视角吐槽，不要有任何劝解、安慰等环节。这是点评节目，不是咨询节目。"明确与情感咨询标签的区分。

### 6. 深夜来信标签 Guide（v520）
