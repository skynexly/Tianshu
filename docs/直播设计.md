# 直播功能设计存档

> 本文档记录直播功能的架构决策、标签池、数据结构与生成逻辑，便于跨会话延续开发。
> 直播挂在「视频 App」的「直播」分类下（`_VIDEO_CATS` 里的 `live`），但底层逻辑是电台的近亲、又比电台灵活得多。

## 一、直播的基因（与电台/点播的本质区别）

直播不是点播（电影/剧集）的变体，而是**电台的近亲但更野**：

| 维度 | 电台 | 直播 |
|---|---|---|
| 形态 | 单向播出，半固定结构（开场/正文/结尾） | 即兴，无剧本无定式，跟着互动走 |
| 单位 | 频道 → 一期节目 | 直播间（类似电台频道） |
| 主体 | 个人主播 | 个人 / 团播为主 |
| 画面 | 无（声音） | **竖屏**为主，UI 不能复用点播的横卡 |
| 互动 | 节目里预埋的环节（投票/点歌锚点，生成一次定死） | **强互动、双向实时**：观众发弹幕/打赏/高亮 → 主播回应 → 观众再发 |
| 经济 | 一次性小额打赏氛围 | 打赏冲榜（金额可以很大）+ 动态打赏榜 |

## 二、两种模式（镜像关系）

```
看别人播（先做）
  · 主播是 AI（功能模型）
  · 用户在直播间发弹幕/打赏/高亮 → 触发主播回应 + 观众弹幕
  · 不碰主线，纯功能模型独立调用
  · 观看历史以「直播间」为单位（类似电台订阅）

用户开播（后做，依赖看别人播的 UI 与数据结构）
  · 主播是用户（主线 AI 演）
  · 观众弹幕是 AI（功能模型）
  · 用户在主线里回应弹幕 → 碰 chat.js 的 _onDone 主线回合钩子
  · 双调用（主线一次 + 手机一次），需防主线自己编弹幕，重要弹幕回灌主线
```

两者共用：同一套竖屏 UI、同一套弹幕渲染、同一套直播间数据结构与打赏榜。只是「谁是主播、谁调 API」对调。**这是先做「看别人播」的核心理由——它建的 UI/数据结构能被「用户开播」直接复用。**

## 四、标签池 `_LIVE_TAGS`（已定稿，10 个）

> desc 字段在预览生成阶段拼成 `- 标签名：描述` 喂给 AI，作用是让 AI 判断挂哪个标签 + 据此生成符合调性的直播间预览。
> 长尾离奇类型（修女告解室 / 抓娃娃直播 / 树洞等）走「用户自定义标签」兜底，不预设硬塞。

1. **游戏直播**：主播打游戏给观众看，边玩边唠——可以是技术流硬核操作、欢乐整活、代打、代抽卡、剧情向流程、电竞二路解说，也可以是和观众连麦开黑。涵盖各类游戏直播。主播可以是真人主播，也可以是套着皮套的虚拟主播。

2. **单人唱歌**：一个人的唱见直播间，主播弹唱/清唱/点歌台，观众点歌、刷礼物、聊天互动。主播可以是真人歌手，也可以是套着皮套的虚拟主播。

3. **唱歌团播**：多位主播一起的歌房/团播间，轮流唱、合唱、battle、互相起哄接梗、陪聊，热闹的多人唱见现场。

4. **单人舞蹈**：一个人的舞蹈直播间，主播跳宅舞/街舞/国风舞/热舞/机械舞等，陪伴聊天，观众应援、点舞、刷礼物。可以是真人，也可以是套皮套跳舞的虚拟主播。

5. **舞蹈团播**：多位主播一起的舞蹈团播间，齐舞、轮播、battle、整活。

6. **带货卖货**：电商带货直播间，主播卖货、讲解商品、上链接、催单、抽奖宠粉，观众下单、问价、参与秒杀。

7. **颜值聊天**：主打颜值和陪伴的聊天直播间，主播露脸唠嗑、和观众互动调侃、读弹幕、玩游戏，氛围轻松暧昧或欢乐。可以是真人，也可以是虚拟主播。

8. **兴趣陪伴**：主播专注做一件具体的事给观众看——画画、做手账、手工、做饭、吃播等，一边做一边碎碎念陪聊，观众围观、催更、点菜、闲聊。

9. **ASMR**：助眠/解压向直播间，主播用各种音效、轻声细语、耳语陪伴观众放松入睡，重氛围和声音质感。可以是真人，也可以是虚拟主播。

10. **答疑咨询**：主播以某种专业身份坐镇，观众抛问题、主播实时解答——涵盖律师普法、情感咨询、心理疏导、MBTI/性格分析、塔罗占卜、命理玄学、职业规划等各类"问一答一"的咨询直播。

## 五、预览生成（A1，仿电台 `_radioGenPreview`）— 已落地

### 调用骨架（`_liveGenPreview(kw)`，phone.js 约 13764）
- 走功能模型 `Settings.getWorldvoiceConfig()`（**非主线模型**），与电台/视频/服装/家具同一套。
- `_phoneJsonArrayWithRetry({ label:'直播间', ... })` 生成 JSON 数组。
- **messages 必须 system + user 两条**（血泪坑：只有 system 没有 user，很多中转站直接 500）：
  ```js
  messages: [
    { role:'system', content: sysPrompt },
    { role:'user', content: kw
        ? `请生成一批贴合「${kw}」的、此刻正在播的直播间。`
        : '请生成一批此刻正在播的直播间。' }
  ]
  ```
- `temperature: 0.95, max_tokens: 6000`（3000 会把 8 间 ×200 字 concept 截断成残缺 JSON；6000 才够）。
- loading：`_liveShowLoading(true/false)` 全屏遮罩 + spinner，catch/empty/success 三出口都要关。

### sysPrompt 已落约束
- **时段约束**：深夜 ASMR/情感、早八打游戏做饭、下午晚高峰、带货黄金档；airTime 要跟时段自洽。
- **tags 严格单选**，从 10 个里挑 1 个，desc 拼成 `- 标签名：描述` 喂 AI 判断挂哪个标签。
- **concept 是重中之重**（四块）：① 主播是谁 ② 内容调性 ③ 招牌玩法/口头禅 ④ 气氛标签。虚构主播的人设全锚在这里，A3 进间不用再查任何外部资料就能稳定还原人设。
- **streamerName 命名规则**：命中下方主播名单的真人**必须用本名**（A3 才能拿本名去查 NPC 真实资料）；虚构主播自由编网名（人设已在 concept 里固定，名字只要好记贴合人设即可）。
- 虚拟主播（套皮套）适当出现但**真人为主、不要过半**。
- 世界观禁现实专有名词。
- 保密红线：不暴露 AI / 提示词 / 生成 / 模型等系统词汇，全程真实直播平台口吻。
- 字段示例里**不要用引号包裹值**（如 `每晚20点` 而非 `'每晚20:00'`），防模型照抄出非法 JSON。

### NPC 接入——已实现（决策从「延后」改为「已接入」）
- **链路**：世界观编辑 → 视频区 → 「主播名单」按钮 → `openVideoCastEditor('streamer')` → 勾选存进 `w.phoneApps.video.streamerWhitelist`。
- worldview.js `_VIDEO_CAST_ROLES.streamer.key = 'streamerWhitelist'`，与 phone.js `_liveBuildCastBlock` 读取的字段名一字不差；保存走 `_videoToggleCastNpc` 用 `_VIDEO_CAST_ROLES[_videoCastRole].key`，链路验证通畅。
- `_liveBuildCastBlock`（phone.js 约 13732）读 `w.phoneApps.video.streamerWhitelist`：
  - **空名单 → 返回 ''（全虚构）**，这是默认设计意图，不是 bug。
  - 有勾选 → 拼「可出场主播名单」段 + 每人人设详情（每人 ≤160 字），并强调"人设契合度审慎判断、绝不为用真人扭曲角色设定"。
- 要看到真人主播，用户须先在世界观编辑里勾人。

> **三套现有参照对比**（决策依据）：
> | App | 机制 | 默认行为 |
> |---|---|---|
> | 电台 | castMode 三档（default/disabled/whitelist） | `default` = 全塞，AI 自选 |
> | 阅读 | 纯白名单（无 default 档） | 空 = 全虚构 |
> | 视频 | 纯白名单 × 7 个角色维度 | 空 = 全虚构 |
>
> **定论：直播跟视频/阅读这一派——纯白名单、默认空 = 全虚构。** 电台默认全塞之所以可行，是因为电台 NPC 只本色出镜、门槛极低；直播的团播齐舞/套皮套门槛高得多，默认全塞会把剧情角色乱塞进团播。未来如需按品类拆白名单（团播/套皮套类单独一个更严的白名单），是后话——但注意直播白名单目前**并入视频命名空间**，按品类拆要重新设计。

### 真名 vs 网名——关键范式（照搬电台，不拆字段）
- 走过的弯路：曾想拆 `streamerRealName` + `streamerName` 双字段，被"电台也大部分是虚构的"推翻。
- 电台真正的范式：**本名做身份键，系统按本名查网名显示；虚构 dj 只有一个名（AI 编什么是什么）**。电台虚构不拆字段能不出问题，是因为电台一期节目一次性生成完，dj 名只需期内自洽。
- 直播 A3 是另一次调用，但**有 concept 兜底虚构人设**，所以同样不拆字段。
  - **存的**：`streamerName` = 本名（真人）或网名（虚构）。
  - **显示的**：前端 `_liveFeedPageHtml` 用 `_ltGetDisplayName(streamerName)` 转 `displayName`——真人查到 NPC 的 onlineName 就显示网名，虚构主播查不到原样显示编的网名；头像首字母跟 `displayName` 走。
  - `_ltGetDisplayName`（phone.js 约 4395）：同步函数，先查 chatContacts.nickname，再查 NPC.onlineName。
- **A3 铁律**：传给 AI 的提示词要用 `streamerName`（本名），**不能用 displayName**，否则命中真人时 AI 拿网名对不上 NPC 资料。

### 预览 JSON 字段（已定稿）
存进 `pd.videoDiscover['live']`，**整批替换（不累积）**——直播是即时流，刷新即换一批此刻在播的，不留历史。批内按 `streamerName + title` 去重。
```js
{
  id,           // 'live_' + Date.now().toString(36) + random
  kind,         // 'live'
  title,        // 直播间标题 / 开播主题（简短有钩子）
  tags,         // 品类标签（从 _LIVE_TAGS 选 1 个）
  isGroup,      // 是否团播（唱歌团播/舞蹈团播为 true，其余按内容判断）
  concept,      // 核心概念（≤400，用户不可见，A3 进间生成时用）
  intro,        // 直播间简介（≤60，用户可见）
  streamerName, // 主播名（≤30，真人=本名 / 虚构=网名）
  airTime,      // 开播时段描述（≤40，与时段自洽）
  onlineHint,   // 在线人数氛围数字（≤20）
  coverKeyword, // 封面画面描述（≤80，存但不渲染，留作以后接图用）
  cover,        // 封面图（留空，无图时纯渐变背景）
  createdAt     // Date.now()
}
```

## 六、打赏与经济（后续阶段，已厘清架构）

> **重要架构铁律**：手机里所有购物/打赏行为都是**前端即时扣余额 + 即时记账 + 日志告诉主线"已扣完别管"**（参照阅读打赏 `_readingShowRewardSheet`、服装/家具商城、转账）。直播打赏照此办理——不走"日志让主线扣"，也不存在延迟结算。
>
> （已清除：`flushActionLog` 里那段过时的"购物合计→请扣除此金额"逻辑，它和前端即时扣费冲突会导致双重扣款。phone v898）

打赏冲榜 = 三个独立小机制：
1. **填金额面板**：选礼物 or 直接填金额 → 前端即时扣 + 记账 + 写日志告诉主线已扣
2. **打赏榜**：开播时功能模型生成初始榜（一批 NPC + 累计金额，符合直播间体量）；用户打赏即时累加重排；NPC 金额可随互动微调（后续增强）
3. **金额动态**：榜单数据存活在直播间会话对象，本地累计 + 重排序

## 六·五、A3 进直播间（接 AI 真生成）— 已落地

> 进直播间不再用假数据，调功能模型一次性生成"此刻这一整轮正在发生的现场实况"。

### 核心函数（phone.js）
- `_liveGenRoom(w)`（约 14210）：详情生成主函数，返回同构对象 `{ acts:[{type,text}], danmu:[{at,name,text,highlight}], rank:[{name,amount}] }`，失败返回 null。
  - 走 `_phoneJsonObjectWithRetry`（label:'直播实况', temperature:0.95, max_tokens:6000）。
  - 末尾 `_stripMainlineArtifacts` 清洗 + 兜底校验（acts/danmu 空则判失败、at 限 0~1、danmu 按时序排、rank 切前三）。
- `_liveStreamerProfile(streamerName)`（约 14155）：用 `_collectChatCandidates()` 按名字精确匹配，命中真人 NPC 返回「本场主播真实身份资料」块（≤400 字），让 AI 严格按 ta 人设演。
- `_liveBuildInject(w)`（约 14173）：**详情/回应共用的公共注入块**——`_readingWorldviewBlock(false)`（世界观基础+地区势力速查+节日+知识，**不含 NPC**）+ 主播身份块 + 主线（轻话术）+ 面具块（开关控制）。
- `_liveEnterRoom(id)`（async，约 14313）：`_liveShowLoading(true)` → `await _liveGenRoom(w)` → `_liveShowLoading(false)`；失败 toast 退出、不建 overlay。

### 注入配方（喂什么，经多轮拍板定稿）
- **世界观**：`_readingWorldviewBlock(false)`——精准产出"世界观基础 + 地区势力速查（只发 summary 简介）+ 节日 + 知识"，**完全不含 NPC**（参照电影那边的精细控制）。预览/详情都用它，**无条件发**。
- **NPC**：详情里只有【主播命中 NPC】单独抓详细资料（`_liveStreamerProfile`），其余 NPC 不发（避免速查表和详细资料对立）。
- **主线**：`_buildRecentPlotBlock()` 无条件发，包"仅供把握当下氛围、除非主播本人是当事人否则别直接提"的轻话术（参照电台/新闻的"保密协议 + 主线降存在感"，**不是**阅读的"相处模式参考"那套）。
- **面具**：`_liveTellIdentity()` 开关控制（设置页"告诉主播我是谁"，默认开，`pd.liveTellIdentity !== false`）。开则发用户面具背景信息，**但强制声明"这是背景信息不是让你替 ta 发言"**。

### 设置页面具开关
- `_liveOpenRoomSettings` 全屏页加"观看设置"段，开关行"告诉主播我是谁"。
- 读：`_liveTellIdentity()`（同步 `_getPhoneDataSync`）；写：`_liveSetTellIdentity(on)`（异步 `_savePhoneData`）。

### sysPrompt 三条 AI 越界禁令（血泪教训：AI 会拿面具替用户发弹幕、把主播写进榜单）
1. **面具块**：这是背景信息不是让你替 ta 发言；所有弹幕只能是其他观众；不冒用用户名义；主播也不点名回应"用户本人"。用户的弹幕只由用户亲自发出。
2. **弹幕（含高亮留言）**：发言人绝不能是主播本人、不冒用主播名字。
3. **打赏榜**：观众打赏排名，绝不出现主播本人（主播不给自己打赏）。

## 六·六、A4 用户互动 — 已落地（常规流程）

> **架构铁律（必须铭记）**：
> 1. **直播间是持久累计**，跟随关注体系（关注就存），**不再"刷走即焚"**。`w.mySpent` 跟着直播间走，关注后下次进来累计还在。
> 2. **所有互动走"攒批手动刷新"模式**（仿电台连线 `_radioCallRefresh`）：用户发弹幕/打赏/高亮一次性做完，点刷新才把这批打包发给主播。AI 不主动响应、不实时追加。
> 3. **扣钱铁律**：前端即时扣余额 + 即时记账（`_addLedgerEntry`），**上屏扣钱都算数**（哪怕没刷新就退出，钱已扣、榜单累计已更新）。

### 数据结构
- `w.mySpent`：用户在这个直播间的【累计打赏金额】，持久落盘到 `pd.videoDiscover['live']`（`_liveSaveRoomState`）。
- `_liveRoomCtx`：加 `myActions`（pending 队列，攒未刷新的动作）+ `busy`（刷新锁）+ `room`（当前 room 引用，重渲榜单用）。

### 三个互动入口
- **弹幕**（输入框去 readonly，回车发）：免费 → 立即上屏（右侧主色气泡 `.phone-liveroom-dm.me`）+ 写历史 + push pending。
- **打赏**（gift 按钮 → `_liveShowRewardSheet`）：面板标题「打赏主播」，礼物档位 `_LIVE_GIFTS`（汽水6/鲜花66/应援棒99/小火箭188/比心520/守护666/钻石999/星河1314，金额沿用阅读、名字改直播场景），**加数量输入框 + 实时算合计**，选货币扣钱 → 累计 + 重渲榜 + 上屏 + push pending。
- **高亮**（star 按钮 → `_liveShowHighlightSheet`）：固定价 `_LIVE_HIGHLIGHT_PRICE = 30`，填留言 → 选货币扣钱 → 高亮上屏 + push pending。

### 刷新 + 主播回应
- **刷新按钮**：顶部关注按钮左边（`.phone-liveroom-refresh`，busy 时转圈）。
- `_liveRefresh()`：busy 锁 + pending 为空提示先互动；打包 `myActions` → `_liveGenRoomReply` → 成功续播、失败退回 pending（仿电台连线）。
- `_liveGenRoomReply(w, batch)`：注入同详情（`_liveBuildInject`）+ 整场历史最近 30 条（`_liveHistoryToText`）+ 用户这批动作 + 进前三时告诉主播"榜上有 ta"。返回追加的 `acts + danmu + rank`。
- `_livePlayReply(reply, streamerName)`：清旧定时器、逐条续播主播 acts + 新观众弹幕、写历史；**用回应里的新 rank 替换 `ctx.room.rank` 重渲榜单**（路人榜会动）。

### 主播回应守则（提示词重点）
1. **最重要：主播先做本职**——该打游戏打游戏、该唱歌唱歌，回弹幕是穿插在直播过程里的，不能为回弹幕丢下正事。
2. **不要只回用户一个人**——回用户的同时，也要像真主播那样念出并感谢其他观众的弹幕、别人的付费高亮、别人刷的礼物。
3. 详情生成的 danmu 要求"至少 3-4 条是观众刷礼物/打赏的弹幕"，让现场一进去就有人在花钱。

### 榜单语义（已定稿）
- 榜单 = **本直播间的粉丝打赏榜**，榜上是这个间的"大哥大姐"（本场打赏最多的粉丝），数字是 ta 在这个直播间的【累计打赏金额】。
- `_liveRankHtml(room, w)`：AI 路人榜前三 + 用户 `mySpent` 混排。进前三 → 榜内对应名次高亮"我"；没进前三 → 前三照旧 + 底部单独挂一行"第 N 名·我·累计"。
- **累计 vs 单次的定论**：我们的直播间是持久实体（不是一场场开播下播），所以 `mySpent` 是**总贡献榜**语义（永久累加、不清零），和"持久累计、关注就存"的整体架构自洽。提示词里明确"累计打赏金额"。
- **PK（A5）是独立计数**：真实直播 PK 算的是"限时窗口内的增量"（这一局清零），跟常驻粉丝榜（总累计）是两套。未来做 PK 时新建临时 `pkSession`（记本局开始时间 + 双方 PK 值 + 我这局助攻额，从 0 算），打赏时 `w.mySpent`（粉丝榜）和 `pkSession.myContribution`（PK 助攻榜）双计、互不干扰，PK 结束清掉 session。这正好印证常驻榜用总累计的合理性。

## 七、预设主播 + 关注 + 随机开播（已讨论，暂缓，A3 之后再做）

> 这一节是 A3 详情做完之后的方向存档，**当前不做**。直播间实体到时直接复用电台那套（电台已是"实体持久 + 随时进"的模式）。

### 起点纠结与定论
- 反复纠结的死结：「即时流刷到看命」与「个性化/关注」天生互斥——即时流刷走即焚，而关注/换壁纸头像要求间是持久实体。
- **最终拍板**：
  - **关注 = 持久化**。关注了就把整个直播间实体存下来，用户想看就打开对应直播间，**啥时候点进去啥时候播，不限时、不判时段**。
  - **「启用主播随机开播」开关**：一个全局开关。关（默认）= 关注列表里都显示在播，随时进；开 = 进关注列表时前端给每个主播随机掷一次在播/未播，未播的灰掉，纯前端随机、零 API，用来找补"真人不可能 24h 在播"的不自然感。
  - 个性化（换壁纸/头像）跟着"持久实体"走，关注体系做了自然就有了。

### 三类对象（厘清，别再搅在一起）
1. **AI 纯虚构的临时间**：发现页即时流，刷到看命，刷走即焚，不能关注/个性化。已实现。
2. **预设主播 `_LIVE_PRESETS`**：人设/壁纸/头像**写死在配置**（用户填，非运行时换），按 `period` 时段标记置顶进发现页，点进去 A3 现生成。这条不受"AI 时间格式不可控"影响——period 是我们自己写的枚举，100% 可靠。
3. **真人 NPC 主播**（命中 `streamerWhitelist`）：避免"每次必现"，未来改成按概率/时段混入即时流。

### 时间问题的定论（重要）
- AI 生成的 `airTime` **没有固定格式**，靠正则解析做"开播门禁"守不住，**砍掉门禁这个想法**。
- 直播是即时流，"刷出来的=此刻正在播"，所以 `airTime` 只是**氛围注脚**，不拦截、不解析。
- sysPrompt 的时段约束**保留**，但它只管"让 AI 生成贴合当前时段氛围的内容"，是创作提示，不是门禁。
- 唯一可靠的时段判断用在**预设主播 period**：`Calendar.parseAbsoluteTime(_getGameTime()).hour` 拿小时（兜底 `new Date().getHours()`），跟写死的枚举比，无解析风险。

### 实现要点（待做时参考）
- 预设池常量 `_LIVE_PRESETS`：每项 = 完整直播间对象 + `period`(morning/day/evening/night/all) + `presetAvatar` + `cover`，图放独立目录 `img/live/`（不混进视频随机池 `img/covers/`）。
- 关注持久桶：单独存（如 `pd.videoFollowedLive`），存**整个间快照**（不只是 id，因为即时流那批会被整批替换掉）。
- 关注入口：A3 直播间内「关注」按钮存快照；关注页（直播底部 tab 或视频「我的」）列出。
- 点进关注的间：A3 现场生成，concept 是存下来的 → 人设永远一致、内容每次新鲜。
- 随机开播骰子时机待定：A=每次打开关注列表掷一次（最真实，但退出再进可能变下播）/ B=进一次 App 锁定一次。
- 视频封面存法可复用：相对路径 `img/covers/cover_xxx.jpg`，前端拼 `_VIDEO_COVER_DIR + file`；预设主播同理用 `img/live/`。
- 顺序建议：**预设主播 → A3 → 关注体系**（关注点进去依赖 A3 的现场生成能力）。
