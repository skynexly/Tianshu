# 已知问题与修复记录

## 1. 切主题后进编辑页空白（陈年 bug · 已用自愈兜底）

### 症状
在「设置 → 界面主题」手动切换主题后，回到聊天，再进「世界」面板点开
**世界观 / 单人卡（角色）/ 世界书** 任意一个进入编辑页，编辑页内容区整个空白，
**只剩顶部标题和 Tab 按钮**。必须手动刷新一次才能恢复（数据没丢，纯渲染问题）。

### 关键特征（用来区分真假）
- **手动切主题会触发**；**世界观绑定主题导致的自动切主题不会触发**。
  - 自动切走 `Worldview._applyBoundTheme`，只调 `Theme.apply()`。
  - 手动切走 `Theme.applyPreset` / `activateCustomPreset`，**多调了一个 `Chat.renderAll()`**。
- 三个编辑页会一起中招，因为它们都是「先 `switchEditTab('basic')` 再填表单」的同构结构。
- 刷新能恢复（PWA 的 JS 走网络优先 `no-store`，刷新必加载最新代码，且清掉了切主题污染的内存/渲染态）。

### 根因（机制）
切主题后，编辑表单加载时 **tab 内容容器的渲染高度变成 0**（CSS 可见性层面，
不是 JS 抛异常——给 `_loadEditForm` 加 try/catch 后仍空，已排除抛错中断）。
表现就是 `wv-edit-tab-basic` / `sc-edit-tab-basic` 这类内容区 `offsetHeight = 0`，
看起来只剩标题和 Tab。

> 历史：v688 系列修过多轮，都在动 `withThemeFade` 的淡入动画（给 `#app` 设残留
> inline opacity/transform）。v688.2 直接去掉了 fade 动画一度治好；做「状态栏美化助手」
> 后复发。**真正的隐藏源至今未最终定位**（见下方「待办」）。

### 当前修复（症状层自愈，已生效）
在三个编辑页的加载入口，进面板后用 `requestAnimationFrame` 检测一次：
若 basic tab 内容区 `offsetHeight === 0` 或带 `hidden`，就**强制清掉从内容区到面板根
整条祖先链上的隐藏样式**（opacity/transform/display/visibility/height/max-height + `hidden` 类），
然后重新 `switchEditTab('basic')`。这样不用手动刷新就能自愈。

涉及文件与函数：
- `js/worldview.js` → `_loadEditForm()` 末尾（世界观 + 世界书共用此函数）
- `js/single_card.js` → `_openEditPanel()` 末尾（角色 / 单人卡）

相关版本：`worldview.js?v=26`、`single_card.js?v=9`、`ui.js?v=4`

### 顺带做的健壮性兜底
- `worldview.js` `switchWorldTab('wv')` 分支补 `load()`（原来只有 char/lb 重渲，wv 不重渲）。
- `ui.js` 进 `worldview` 面板时 reset 到 wv Tab。
- `_loadEditForm` 数据填充段包 try/catch，单字段失败不再连锁中断，并 `console.warn` 记录。

### 待办（把根因也堵死，可选）
自愈是症状层兜底，真正「谁把内容区高度变 0」还没挖出来。已在世界观面板留了诊断探针：
修复时把空白前各层 DOM 的 `高度/opacity/display/visibility` 写进
`panel-worldview-edit` 的 `data-blank-probe`。复现空白后，用地址栏执行（注意手动补回开头的 `javascript:`）：

```
javascript:alert(document.getElementById('panel-worldview-edit').dataset.blankProbe)
```

把弹出的字符串记下来，就能定位是哪一层、被什么属性隐藏的。根因修掉后可移除自愈 + 探针代码。
