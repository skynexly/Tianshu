# 文档生成世界观设计

> 状态：**方案讨论中，未实装**。开售稳定后再做，时间充裕。
> 定位：用户上传一份自己整理的世界观资料（可能很长，实测超长文档可达 17w token），系统把它拆解、归类成几个「资料桶」，之后各分模块 AI 生成（节日/知识/地区/NPC/属性/任务……）**只读自己相关的那一桶**，而不是每次都塞全文。
> 一句话：**map-reduce 处理超长文档——切分只付一次成本，之后各模块反复引用小桶。**

## 一、要解决的核心矛盾

- 用户想「扔一份文档进去，AI 照着它写世界观」。
- 但文档可能极大（17w token）：
  1. 大多数模型窗口装不下（DS/便宜中转站常见 6w–12w）。
  2. 若做成「每个模块生成都塞全文」，成本 = 17w × N 个模块，重复消耗巨大（用户自己的 API，钱是用户的，但没必要这么烧）。
  3. 全文里大部分是无关噪音，AI 抓不住重点，生成质量反而差。
- **摘要方案（压成几千字精华）会丢细节**，被否。
- **选定方案：分桶切分**——保留原文细节，按用途归类，各取所需。

## 二、桶的划分

对齐现有分模块生成能力（见 worldview.js 的 `_doAiGen*` 系列），初步分 5 桶 + 1 兜底：

| 桶 | 内容 | 服务的生成模块 |
|---|---|---|
| 基础设定桶 | 世界背景、地理、历史、种族、力量/科技体系 | 世界观描述、设定、开场；全局属性 |
| 玩法桶 | 属性体系、任务/成长、经济规则、货币 | 全局属性、角色属性、任务阶段、商店 |
| 地区势力桶 | 地图、区域、组织、派系、阵营 | 地区、势力（NPC 归属） |
| 人物桶 | 关键 NPC、重要角色 | 全局 NPC |
| 文化桶 | 节日、习俗、历法、杂项知识、黑话物产 | 节日、动态知识、历法 |
| 未分类桶（兜底） | AI 无法明确归类的内容 | 供用户手动调整 |

> 一段内容可同时进多个桶（既像设定又像玩法时）。允许重复，不强求互斥。

## 三、切分流程（map-reduce）

### map 阶段（分批打标签）
1. `Utils.readFileAsText(file)` 读出全文（已支持 txt/md/json/docx/pdf，限 5MB —— **注意：5MB 限制对 17w 文档可能要放宽或分文件**）。
2. 全文按段落/字数切成若干块（每块控制在单模型窗口安全范围内，比如每块 4k–8k token）。
3. 每块单独调 AI：「这段内容分别属于哪些桶？」→ 返回该段的桶标签（可多选）。
4. **切分本身也绕不开「读一遍全文」的一次性成本**，但只付一次，且分批不会爆窗口。

### reduce 阶段（归拢）
1. 按桶标签把各段原文归拢，拼成每个桶的完整文本。
2. 每桶可再做一次可选的「去重/整理」（把散落同类内容合并），此步可省以控成本。
3. 切分结果存进世界观对象（见第五节）。

## 四、各模块注入改造

现有每个 `_doAiGen*` 的 `userMsg` 结构固定为：
```
## 用户额外要求（可选）
## 世界观设定（baseCtx）
请生成 N 个 XX，严格按要求输出 JSON。
```
改造：在 `baseCtx` 之后、插入一块 `## 参考资料`，内容 = **该模块对应的那一桶**（不是全文）：
- 生成节日 → 注文化桶
- 生成地区/NPC → 注地区势力桶 / 人物桶
- 生成属性/任务 → 注玩法桶
- 生成描述/设定 → 注基础设定桶

改动模式很干净：只在现成拼接里多插一段字符串，**不动任何 JSON 解析、不碰输出结构**。

## 五、数据结构（初稿）

存进世界观对象，例如 `w.docBuckets`：
```
docBuckets: {
  _sourceName: '原始文档名',
  _createdAt: 时间戳,
  _rawTokenEst: 170000,       // 原文估算 token，给用户成本提示
  base:     '基础设定桶文本',
  gameplay: '玩法桶文本',
  region:   '地区势力桶文本',
  npc:      '人物桶文本',
  culture:  '文化桶文本',
  unsorted: '未分类兜底文本'
}
```

## 六、三个硬骨头（实装前必须想清）

1. **切分那步要读全文**——绕不开的一次性成本。靠分批 map 避免爆窗口。用的是用户自己的模型/API，**成本由用户自负**，我们只需在切分前如实提示大致消耗，用户自己掂量要不要跑。
2. **成本对比（仅供用户参考）**：旧方案「每模块塞 17w」= 17w×N；本方案「17w 读一次 + 之后各读小桶」= 17w + 小桶×N。本方案省得多，但这是用户自己的账，我们不代做省钱决策，只做知情提示。
3. **切不干净**：AI 分类有模糊地带。对策——允许一段进多个桶 + 保留未分类兜底桶 + **让用户能在 UI 手动调整切分结果**，不全指望 AI 一次切对。

## 七、UI 设想

- 新建/编辑世界观页加入口：「从文档生成」。
- 上传 → 显示解析进度（分批切分是慢操作，要有进度条/toast）。
- 切分完成后展示各桶内容，**允许用户手动编辑/搬运段落**。
- 之后各模块的 AI 生成弹窗里，若该世界观已有 docBuckets，自动带上对应桶（可开关）。

## 八、实装步骤（预估，分批做）

1. 放宽/处理文档大小限制（17w 场景）。
2. 分批读取 + map 打标签 + reduce 归拢的切分内核。

4. 切分结果的展示/手动编辑 UI。
5. 逐个 `_doAiGen*` 注入对应桶（工程量集中在此，可分批）。
6. 成本提示与进度反馈。
