# 角色记事本设计

> 状态：**设计阶段（未实装）**。本功能等电台、邮箱两个 App 都做完后再开始落地。
> 本文档只记录 0 号已拍板的决策与待裁决的空白点，不含未经确认的扩展设定。

## 一、定位与动机

电台（及未来的邮箱等功能）会让 NPC 露出主线剧情里还没展现的另一面——某个秘密、某种特质、某段生活细节。这些是这些功能场景里**新生成**的角色侧写。

「角色记事本」是一个**跨功能的统一机制**：把这些碎片场景里 NPC 露的侧面，沉淀成挂在该 NPC 名下的记事本条目，成为这个角色的**追加设定**。之后主线里该 NPC 在场时，记事本内容会被召回注入，让 NPC「带着这些新设定」行动。

这样做从根上消除了原本的割裂感：一旦某段内容被采纳进 NPC 的记事本，它就成了这个角色的一部分，不再存在「电台里有她的身影，但她自己不知道」的矛盾。

## 二、已拍板的决策（0 号确认）

1. **只做实名总结**。主播 / 嘉宾这类 NPC 实名参与的内容才进记事本。匿名素材（`connectMaterial` 改写的瓜）**暂不处理**——匿名定位是技术难点（见附录），后续再议。
2. **AI 总结按条追加，绝不覆盖**。每次总结新增条目，旧条目永远不被 AI 改写或删除。
3. **每条玩家可编辑**。玩家能逐条修改记事本内容，编辑后的内容是正式版本。
4. **AI 总结进「待采纳」，玩家确认后才入正式记事本**（防自动+手动打架，见第五节）。
5. **主线中在场 NPC 命中 → 自动注入该 NPC 的记事本内容**。
6. **命中机制 = 关键词命中 + 随机**，复用小纸条（memory.js）的双路评分用法。
7. **统一机制**，电台是第一个数据源，未来邮箱等功能同样接入。
8. **本功能可以做在记忆模块里**（倾向把记事本作为一种记忆类型，复用现成召回逻辑）——存储方案待定，见第四节。

## 三、数据来源：实名 NPC 怎么定位

- **节目侧字段**：`prog.dj`（主播名）、`prog.guest`（嘉宾名），均为明确角色名。
- **命中世界观 NPC**：现有 `cast` 注入逻辑（phone.js ~10282 行）已会拿 `dj/guest` 去世界观 NPC 库精确命中；命中成功即可拿到该 NPC 名，作为记事本的归属键。
- **未命中**（虚构化名 / 单人卡角色）：不生成记事本条目（避免给不存在的 NPC 造档）。

> 匿名素材（`_radioConnectMaterialBlock`）抽中的角色 ID 在生成后即丢弃、正文也按匿名要求抹掉所有身份标识，系统无法回溯归属，故**匿名不入记事本**（决策 1）。

## 四、数据结构（两方案，待 0 号裁决）

记事本归属键 = NPC 名（与世界观 NPC `name` 对齐）。

**方案 A — 作为新的记忆类型（倾向，呼应决策 8）**
- 在 memory.js 里新增 `type: 'charnote'` 的记忆条目。
- 字段建议：
  ```
  {
    type: 'charnote',
    npc: '角色名',          // 归属键
    detail: '一条侧写内容',  // 玩家可编辑
    source: 'radio',        // 来源：radio / mail / ...
    sourceRef: '节目/邮件标识', // 溯源用
    status: 'pending' | 'adopted', // 待采纳 / 已采纳
    scope: '角色卡ID',       // 复用现有 scope 隔离
    createdAt: 时间戳
  }
  ```
- 优点：直接复用小纸条的双路评分召回（在场角色精确匹配 + 关键词命中 + 随机补足），改动集中在记忆模块，天然支持「按条追加 / 编辑 / scope 隔离」。

**方案 B — 挂在 NPC 世界观档案上**
- 给 NPC 条目加 `notes: []` 追加层，与 `detail` 并列。
- 优点：和角色档案绑得最紧；缺点：召回逻辑要另写，且世界观条目结构改动影响面更大。

> 倾向 A。请 0 号确认走 A 还是 B。

## 五、写入流程（AI 总结 → 待采纳 → 玩家采纳）

为避免「AI 自动写」和「玩家手动编辑」互相覆盖，采用两层：

1. **AI 总结**：一期节目结束后（实名 NPC 命中时），AI 总结出这期里该 NPC 体现的侧写，**按条**写入「待采纳区」（`status: 'pending'`）。
2. **玩家审阅**：玩家在记事本 UI 看到待采纳条目，可编辑、可丢弃。
3. **采纳**：玩家确认后条目转为 `adopted`，进入正式记事本。
4. **追加语义**：每次总结只新增 `pending` 条目，**绝不修改 / 删除已有任何条目**（决策 2）。正式记事本只有玩家能动，AI 不直接写正式区。

## 六、读取与注入（主线在场命中）

复用小纸条的双路评分思路（memory.js `retrieve` + 注入位置 chat.js ~1116 行）：

- **取在场 NPC**：`NPC.getPresentNPCs()`。
- **召回**：对在场 NPC 名做精确匹配，命中其 `adopted` 记事本条目；条数不足时按关键词命中 + 随机补足，整体限 3-5 条（与小纸条一致，防上下文膨胀）。
- **只注入 `adopted`**，`pending` 不进主线。
- **注入文案**：作为该角色的「补充设定 / 侧写」提示，让在场 NPC 自然体现，不强制复述。
